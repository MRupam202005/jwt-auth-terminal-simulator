What is a JSON Web Token?

Stateless Authentication

Instead of the server maintaining user sessions in database/memory, JWT stores the session state inside the client token. The server verifies it cryptographically.

Header (Red)

Tells the server which algorithm is used to sign the token (e.g. HS256, RS256). Identifies the content type as a JWT.

Payload (Magenta)

Contains the "claims" (statements about the user, like user ID, username, roles, and issue/expiration timestamps). Note: Not encrypted, just encoded!

Signature (Teal)

Calculated by combining the encoded header and payload with a secret key. Ensures that if any part of the header or payload is modified, the signature becomes invalid.

Access Tokens vs. Refresh Tokens

Why do we use two different tokens instead of just one? Let's check the comparison below.

Feature Access Token Refresh Token
Primary Purpose Authorize API requests directly Request new access tokens without prompt
Lifespan Very short (e.g., 5 - 15 minutes) Long-lived (e.g., 7 - 30 days)
Usage Sent on every HTTP Request header Sent only when the access token expires
Storage (Best Practice) In-memory variable (ideal) or localStorage Secure HTTP-Only Cookie or DB-backed session
Compromise Risk Low (expires rapidly, limiting hacker access) High (requires strict server rotation/checks)

Security FAQ

Q: Why do Access Tokens expire so quickly?

A: Because JWT is stateless. Once issued, a JWT cannot be easily revoked before expiration without building a database blacklist (which negates the benefit of stateless tokens). Making access tokens expire in minutes ensures that if a token is stolen (e.g., via XSS), it becomes useless almost immediately.

Q: How does Token Rotation work with Refresh Tokens?

A: When the client uses a Refresh Token to request a new Access Token, the server can issue a new Refresh Token as well, revoking the old one (Refresh Token Rotation). If a stolen refresh token is reused, the server detects the reuse and invalidates the entire family of tokens, forcing immediate logout of the user.

Q: Where should we store these tokens?

A: For high security, keep the Access Token in client memory (JavaScript variables) and the Refresh Token in a secure HTTP-Only cookie. This protects the Refresh Token from XSS attacks and the Access Token from CSRF, achieving the best of both worlds.
